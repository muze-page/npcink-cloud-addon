# Cloud Addon 后台概览与 0.1.3 发布复盘

状态：已完成

日期：2026-07-13

范围：`npcink-cloud-addon` 的 WordPress 后台信息架构、权益摘要、
站点知识库用量、本地化、视觉一致性和 `0.1.3` 发布闭环。

本文归纳本轮连续界面审查和发布工作的开发思路。当前后台规范仍以
[`admin-surface-standard.md`](admin-surface-standard.md) 为准，WordPress.org
版本与 SVN 修订仍以
[`wordpress-org-release-and-translation-log.md`](wordpress-org-release-and-translation-log.md)
为准；本文不建立新的运行时、权益或发布真相。

## 1. 起点与核心矛盾

后台最初的问题不是缺少信息，而是信息没有按使用频率和操作价值分层：

- 常用状态、低频参数、原始状态和排查说明同时出现；
- 同一事实可能在概览、详情和排查区域重复展示；
- 空缓存显示为“尚未刷新”，却要求用户自行寻找刷新入口；
- 固定英文、Cloud 返回状态和中文界面混排；
- 点数与知识库文章分别使用“剩余”和“已用”，比较成本高；
- 原生进度条受浏览器样式和内容宽度影响，难以稳定对齐；
- 局部按钮、嵌套层级和缩进不一致，放大了页面的复杂感。

因此，本轮目标不是继续增加信息，而是让默认页面只回答三个问题：

1. Cloud 连接和套餐现在是否可用？
2. 最常用的额度还剩多少？
3. 当前是否存在需要管理员处理的问题？

## 2. 信息分层决策

### 默认概览

概览只保留高频、可快速判断的信息：

- 套餐名称与可用状态；
- 可用点数；
- 站点知识库投递开启时的可用文章额度；
- Cloud 明确返回时的运行额度；
- 仅在异常或积压存在时出现的监控、投递提醒；
- WordPress AI、站点知识库投递等本地授权。

概览不重复成功时间戳、原始枚举、策略字段、完整使用明细或 Cloud
服务面板数据。用户需要详细解释时，通过“查看服务详情”进入
“高级与排查”。

### 站点知识库

站点知识库文章额度只在本地投递已开启时展示。这样既避免向未启用用户
展示无关数据，也明确区分两种责任：

- WordPress 只拥有是否投递公开内容的本地授权和有界传输；
- Cloud 拥有索引执行、用量真相、生命周期、新鲜度和向量存储。

站点知识库页保留常用刷新、投递状态和有界索引操作入口；技术投递细节
仅在需要恢复或排查时展开。

### 高级与排查

低频参数统一进入“高级与排查”，包括：

- 账户和使用详情；
- 连接检查与最近失败；
- Runtime Runs 的只读状态、结果读取和有界重试；
- 手动连接恢复；
- 仅在有操作价值时出现的缓冲、错误和技术细节。

这一区域仍是连接器的摘要/详情面，不是第二个 Cloud 控制台。

## 3. 自动刷新与失败恢复

打开插件后，不应让用户面对空白的“套餐与权益”再去寻找顶部刷新按钮。
最终采用以下策略：

- 状态为 `not_refreshed` 或 `stale` 时自动发起一次后台读取；
- 请求期间展示加载状态并阻止重复请求；
- 成功后原位更新套餐、点数、运行额度和知识库文章用量；
- 旧值仍可用但更新失败时保留旧值，并标记“更新失败”；
- 没有可保留数据时显示明确的暂不可用状态；
- 只有失败时才显示“重试”，不常驻全局刷新按钮；
- AJAX 请求继续使用管理员权限检查和 nonce。

这让正常路径成为“进入页面即可看到结果”，手动按钮只承担异常恢复，
避免同时出现顶部刷新、行内刷新和重新验证等重复入口。

## 4. 额度语义与视觉规则

### 统一为“剩余”

可用点数和知识库文章统一采用剩余口径：

```text
35,148 / 38,000    剩余 92%    █████████░
9,401 / 10,000     剩余 94%    █████████░
```

数值区表示“剩余 / 上限”，状态区统一表示“剩余百分比”。具体的已用、
剩余和上限数字放在鼠标悬停提示中。这样两行可以直接比较，不再要求用户
先判断一行是“剩余”、另一行是“已用”。

### 统一称为“点数”

Cloud 契约和英文源码可继续使用稳定的 `credit` / `credits` 字段与消息
标识，但 zh_CN 用户界面统一显示“点数”。不要为了界面词汇修改 Cloud
数据契约，也不要把英文单位拼接到中文数值后面。

### 分段进度条

没有直接输出 Unicode 方块字符串，而是使用十段 CSS 遮罩实现同样的
`█░░` 视觉语言，并保留真正的 `progressbar` 语义：

- 数值、状态、进度和操作使用固定网格列对齐；
- 数字使用等宽数字特性，减少刷新时的视觉跳动；
- 进度条通过 `aria-valuenow` 暴露真实百分比；
- 小屏幕改为两列并让进度条独占下一行；
- 知识库用量可按正常、警告和错误状态改变颜色。

相比直接输出字符，这种实现不依赖字体字宽，能表达任意百分比，并兼顾
辅助技术和响应式布局。

## 5. 本地化规则

本轮漏翻译主要来自三类来源：

1. 新增的固定后台字符串尚未进入 zh_CN 维护映射和语言包；
2. Cloud 返回的已知状态（例如 `queued`）直接进入展示层；
3. 套餐名和 `credit` 单位没有经过面向用户的显示映射。

处理规则如下：

- 插件自身固定后台文案使用 `npcink-cloud-addon` text domain；
- zh_CN 兼容映射覆盖关键固定字符串和已知状态；
- `Free`、`Pro` 等已知套餐映射为产品化中文，未知 Cloud 套餐名保持原值；
- `queued` 等已知状态映射为明确的中文操作状态；
- 动态 ability 元数据、provider/model ID、slug 和 contract ID 不在本插件
  翻译，继续由来源系统负责；
- 更新文案后同步 POT、PO、MO，并运行本地化行为测试。

这套边界避免两种错误：一是用户界面夹杂可避免的英文，二是连接器擅自
翻译动态契约内容而制造错误含义。

## 6. 布局与交互取舍

最终保留三项顶层入口：

- 概览；
- 站点知识库；
- 高级与排查。

具体规则包括：

- 同一层级的开关和内容使用一致的左边界；
- 从属开关用容器结构表达依赖，不用随意增加视觉偏移；
- 页面标题、页签标题和区块标题不重复表达同一件事；
- 主操作放在相关状态旁边；
- 长说明进入详情入口，不在默认表格中堆叠；
- 避免嵌套 `<details>` 和过深页签；
- 同一事实只在一个默认位置展示，其他位置链接到其详情所有者。

## 7. 被拒绝的方向

本轮明确没有采用以下方案：

- 在顶部长期放置一个全局刷新按钮；
- 默认展示所有权益、原始状态和排查参数；
- 为每组复杂信息增加独立弹窗或新的顶层页签；
- 同时显示“已用”和“剩余”两套互不一致的百分比口径；
- 用字体字符直接模拟进度条而牺牲对齐和无障碍语义；
- 在 WordPress 中复制 Cloud 的账单、额度明细、索引或运行时真相；
- 翻译动态 ability 元数据和技术标识符；
- 为界面便利新增工作流、队列、审批或 WordPress 写入职责。

这些方案要么增加认知负担，要么形成重复入口，要么越过 Cloud Addon
作为连接器的边界。

## 10. 后续维护准则

以后为后台增加一项信息前，依次判断：

1. 它是否帮助用户完成当前连接、授权或故障恢复任务？
2. 它是否高频，或者是否正在阻塞用户？
3. 它是否已经在其他区域展示？
4. 它是本地连接器事实，还是 Cloud 拥有的服务面真相？
5. 能否用摘要、异常提示或详情链接替代常驻字段？
6. zh_CN 固定文案、对齐、无障碍和失败状态是否同时覆盖？

只有高频或需要立即处理的信息进入概览；低频本地事实进入高级与排查；
Cloud 明细通过链接交给 Cloud；重复且没有操作价值的信息移除。

`0.1.3` 已发布，不直接修改其 SVN 标签。后续缺陷或文案调整应进入新的
Git 提交和 `0.1.4`（或之后版本），重新经过发布门槛后再创建新标签。
